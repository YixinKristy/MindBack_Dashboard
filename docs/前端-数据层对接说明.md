# MindBack 看板 — 前端数据层对接说明

面向：Web 前端开发  
数据源：**Supabase**（PostgreSQL + PostgREST + 可选 Realtime）  
Schema：**`public`**

---

## 1. 数据源在哪里

| 项目 | 说明 |
|------|------|
| **数据库** | Supabase 项目内的 PostgreSQL，`public` 模式下表见下文。 |
| **HTTP API** | `https://<PROJECT_REF>.supabase.co/rest/v1/<表名>`（PostgREST）。 |
| **鉴权** | 请求头需带 `apikey: <SUPABASE_ANON_KEY>`；`Authorization: Bearer <同一 anon key>`（匿名读）或 `Bearer <用户 access_token>`（若已配置登录用户写权限）。 |
| **环境变量（前端）** | `SUPABASE_URL`、`SUPABASE_ANON_KEY`（仅 **anon / publishable**，禁止把 **service_role** 写进前端代码或仓库）。 |

**与旧版快照的关系（过渡期重要）：** 历史上看板曾用单表 `public.kanban_snapshot`（`key='prod'` 的 `jsonb`）存整站数据。当前采用“**双源合并**”过渡：  

- **主数据（真实明细）**：以本说明的规范化表为准（v2 共 9+ 张表）。  
- **编排/归类（周视图、features、inbox 等）**：仍可能来自 `kanban_snapshot` 的 v2 JSON（后续会表化替换）。

## 2. 每个表里放了什么

| 表名 | 用途（给前端的语义） |
|------|----------------------|
| **`phases`** | 阶段（4月/5月/6月/7-8月），用于总览/Backlog 阶段归属与展示。 |
| **`versions`** | 版本管理（iOS/Android/Web 的版本号、状态、提审时间等）。 |
| **`milestones`** | 产品里程碑 / 月度目标（M1–M10），含状态、标签、日期文案、功能说明等。 |
| **`tasks`** | 开发任务：优先级、平台、负责人、状态、类型、来源、是否本周重点、关联里程碑、**关联版本（version_id）** 等。 |
| **`bugs`** | Bug 列表：严重级别、平台、负责人、状态、来源、备注。 |
| **`people`** | 团队成员：姓名（唯一）、角色、工作负载百分比、是否在职。 |
| **`ops_actions`** | 运营动作：周边/联名/活动三类，状态、目标时间（文本）、渠道、负责人、备注。 |
| **`backlog`** | 待规划功能：优先级、平台、来源、备注、**阶段归属（phase_id）**。 |
| **`tech_debt`** | 技术债条目：名称、单个负责人、完成进度 0–100、备注。 |

**软删除：** 各业务表均有 `is_archived`（除 `milestones` 无外，里程碑用 `status` 等表达生命周期）。列表展示建议默认 **`is_archived = false`**（`milestones` 无该字段则全量或按 `status` 筛选）。

**关联说明（v2）：**

- `milestones.phase_id` → `phases.id`（外键，可空）。
- `backlog.phase_id` → `phases.id`（外键，可空）。
- `tasks.milestone_id` → `milestones.id`（外键，可空）。
- `tasks.version_id` → `versions.id`（外键，可空）。
- `tasks.owners` / `bugs.owners` 等与 `people.name` 为 **软关联**（同名匹配），不是数据库外键。

## 3. 要怎么取（查、改、实时）

### 3.1 推荐：Supabase JS

```bash
npm i @supabase/supabase-js
```

```javascript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY
);

// 只读示例：任务列表（未归档，按 id）
const { data: tasks, error } = await supabase
  .from('tasks')
  .select('*')
  .eq('is_archived', false)
  .order('id');

// 带里程碑标题（一次 join）
const { data: rows } = await supabase
  .from('tasks')
  .select('*, milestones ( id, title, status )')
  .eq('is_archived', false);

// 更新一行（需 RLS 允许当前 JWT，见 3.3）
const { data, error: e2 } = await supabase
  .from('tasks')
  .update({ status: 'prog', note: '…' })
  .eq('id', 17)
  .select()
  .single();
```

### 3.2 备选：裸 `fetch`（PostgREST）

```javascript
const base = `${SUPABASE_URL}/rest/v1`;
const headers = {
  apikey: SUPABASE_ANON_KEY,
  Authorization: `Bearer ${SUPABASE_ANON_KEY}`,
  Accept: 'application/json',
};

const res = await fetch(
  `${base}/tasks?select=*&is_archived=eq.false&order=id.asc`,
  { headers }
);
const tasks = await res.json();
```

写入时常用头：`Content-Type: application/json`、`Prefer: return=representation`。

- **PATCH** 单行：`PATCH /rest/v1/tasks?id=eq.17`，body 为 JSON 字段子集。  
- **POST** 插入：`POST /rest/v1/tasks`，body 为对象（勿随意带 `id` 除非你们约定用序列/指定 id 策略）。

常用查询参数（PostgREST）：`eq`、`neq`、`in`、`is`、`order`、`limit`、`offset`、`select` 嵌套关联语法等，见 [PostgREST 文档](https://postgrest.org/en/stable/references/api/tables_views.html)。

### 3.3 权限（RLS）— 前端必须知晓

当前迁移中的默认策略为：

- **所有人可读**：`SELECT` 允许（含匿名）。
- **写入**：策略绑定 **`service_role`**（供服务端/脚本使用）。**Supabase 使用 service_role JWT 时通常会绕过 RLS**，但浏览器端 **禁止** 使用该密钥。

因此：**若要在浏览器内由「已登录管理员」直接 `insert/update/delete`，需要后端同学在 Supabase 为 `authenticated` 角色或指定 `auth.uid()` 增加相应 policy**（与 `kanban_snapshot` 管理员写权限同一思路）。在未加策略前，前端用 anon 只能读、不能写。

### 3.4 实时拉取（可选）

在 Supabase 控制台为表开启 **Realtime**，并在项目中把表加入 `supabase_realtime` publication（按控制台指引操作）。前端示例：

```javascript
supabase
  .channel('tasks_changes')
  .on(
    'postgres_changes',
    { event: '*', schema: 'public', table: 'tasks' },
    (payload) => {
      // 合并到本地 state，或触发重新 select
    }
  )
  .subscribe();
```
